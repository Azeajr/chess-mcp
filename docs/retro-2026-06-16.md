# Retro: Repertoire Analysis Session — 2026-06-16

Tool and workflow shortcomings observed during a live repertoire analysis run.

---

## Issues

### 1. No auto-trigger on repertoire load
The LLM sat idle after `Open PGN` until the user manually typed a prompt. The UI has no hook to fire a chat message when a new repertoire loads. Fixed in this session: `files.ts` now calls `send()` with a standard analysis prompt immediately after `actions.loadPgn`.

### 2. Stockfish evaluates every position with no off switch
The browser Stockfish ran a depth-14 search on every board navigation, including positions the user was just clicking through. No toggle existed. This blocks the engine from doing LLM-requested `evaluate_position` calls and burns CPU on positions that don't need analysis. Fixed: `evalEnabled` signal added to `analysis.ts`; TopBar "Eval On/Off" button gates the `createEffect`.

### 3. `cloud_eval` returns 404 for all non-mainline positions
The LLM called `cloud_eval` before `evaluate_position` on every position. Lichess's cloud DB only covers popular theory; obscure repertoire branches return 404, surfaced as `{ available: false }` to the model. The model then called `evaluate_position` as fallback — doubling tool calls per position. The system prompt does not instruct the model to skip `cloud_eval` for known-obscure lines, and there's no way for the model to predict which positions will be in the DB.

**Fixed:** `cloud_eval` removed from the active tool list (`tools.ts`) and dropped from the system-prompt grounding list (`chat.ts`); `evaluate_position` (local Stockfish) is now the sole, authoritative eval source. The `runTool` case + `cloudEval` import are kept (commented schema) so it can be re-enabled as a fast mainline cache once the doubling/404 noise is solved.

### 4. Tool results invisible in the UI
The chat panel showed `⚙ evaluate_position` chips but hid the actual JSON result. The user couldn't see what data the LLM was working with, whether `cloud_eval` was returning null, or whether an eval came back bad. Fixed: `ChatPanel` now renders `role: "tool"` messages as collapsible `<details>` elements, with error results highlighted in red.

### 5. `MAX_ROUNDS = 6` caps deep analysis
For a repertoire with many branches, the LLM needs one tool round per position checked. At 6 rounds max it can evaluate at most ~5 positions before the loop ends and it must summarize without more data. The limit exists to prevent runaway API costs but is too low for repertoire work.

**Fixed:** `MAX_ROUNDS` raised 6 → 12 in `chat.ts` (≈ one tool round per position checked), still bounded to cap runaway API cost. A "continue analysis" button remains a future option if 12 proves too low.

### 6. In-app LLM cannot reach the MCP repertoire tools (root cause of most of the above)

The Claude Code harness calls tools via the MCP server (`load_repertoire`, `find_repertoire_gaps`, `get_structural_profile`, `analyze_repertoire_congruence`, `batch_review`, `suggest_complementary_lines`, etc. — ~30 tools). The in-app browser chat calls OpenRouter directly with only 5 browser-side tools defined in `apps/ui/src/llm/tools.ts`:

| In-app LLM | MCP (Claude Code only) |
|---|---|
| `get_position` | `load_repertoire`, `load_repertoire_from_file` |
| `get_legal_moves` | `find_repertoire_gaps` |
| `evaluate_position` | `get_structural_profile` |
| `cloud_eval` | `analyze_repertoire_congruence` |
| `propose_line` | `get_repertoire_coverage`, `get_transpositions` |
| — | `batch_review`, `suggest_complementary_lines`, … |

There is no bridge between the UI's OpenRouter client and the MCP server process. The in-app LLM has no way to call `find_repertoire_gaps` or `get_structural_profile`, so it falls back to evaluating individual positions one at a time — exactly the pattern observed in this session.

**This explains issues #3, #4, and #5** above: the model calls `cloud_eval` + `evaluate_position` per position because those are its only analysis tools, hits the `MAX_ROUNDS` ceiling quickly, and never produces the structured gap/structure output that the MCP tools would have given.

**In progress:** design-doc-first. See `docs/design/UI_MCP_BRIDGE_DESIGN.md` — a Vite dev plugin spawns the stdio MCP server and relays it through a same-origin `/__mcp` endpoint (COEP-safe), the browser connects with an MCP HTTP client, and a curated repertoire-tool subset is merged into `tools.ts` with graceful degradation when no bridge is present. No bridge code until the doc is approved.

---

## What worked
- `get_position` correctly returned the full PGN with variations on the first call — the model was grounded without a second round-trip.
- `evaluate_position` (local Stockfish) returned valid evals when called; the eval cache hit rate was not measured but the engine did not crash.
- `propose_line` was not tested this session but the staging mechanism is in place.
