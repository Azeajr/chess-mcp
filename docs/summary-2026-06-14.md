## Phase 1 — New features (AM)

Seven feature issues plus packaging were built by parallel agents and integrated to `main`.
Each shipped with a design doc under `docs/design/`.

### 🆕 `#28` — Persistent eval cache + `cloud_eval`
- **What:** A SQLite-backed, per-position Stockfish evaluation cache (`evalcache.py`) and a new
  `cloud_eval` tool that reads Lichess's pre-computed cloud evaluations. Introduces the shared,
  rate-limited, offline-safe HTTP client (`apiclient.py`) reused by every later network tool.
- **Why:** A repertoire or batch review re-evaluates the same opening positions hundreds of times;
  each call previously spawned a fresh engine and re-searched from scratch.
- **Impact:** Warm-cache lookups return the *same* answer Stockfish would have, with no engine
  spawn; cloud eval gives a free, instant lookup for popular positions. Cache survives restarts.

### 🆕 `#23` — `board_image`
- **What:** Renders a FEN as a base64-encoded SVG, with optional last-move highlight (arrow + square
  tint) and board orientation.
- **Why:** A rendered board is far less lossy for a chat client to reason over than a bare FEN.
- **Impact:** Clients can display positions inline.

### 🆕 `#25` — `lichess_games`, `chesscom_games`, `repertoire_vs_history`
- **What:** Pull a user's public game history from either platform, and cross-reference a loaded
  repertoire against real games (coverage %, average in-book plies, where the user leaves prep, and
  which opponent moves go uncovered).
- **Why:** Connects the repertoire tools to the user's actual practice — the real-world counterpart
  to engine-only gap analysis.
- **Impact:** Turns "is my prep sound?" into "where does my prep actually break down in my games?"

### 🆕 `#26` — MCP App: board browser + PGN stepper
- **What:** An interactive HTML board widget (`boardwidget.py`) served as an `@mcp.resource`, with a
  drag board and a PGN stepper.
- **Why:** Gives MCP clients a richer, interactive surface than tool calls alone.
- **Impact:** Visual exploration inside a compatible client. *(Built + merged; not yet verified in a
  live client — see [Known remaining work](#known-remaining-work).)*

### 🆕 `#30` — `tablebase_lookup`
- **What:** Queries the Lichess Syzygy tablebase for ≤7-piece positions and returns WDL + DTZ +
  best move + category.
- **Why:** Exact endgame ground truth, to study endings and cross-check engine conclusions.
- **Impact:** Authoritative endgame answers where the engine is only heuristic. *(WDL mapping was
  later corrected — see [Bug #1](#bug-1--tablebase_lookup-wdl-mapping).)*

### 🆕 `#24` — `engine_move` (Stockfish / Maia / Leela)
- **What:** A move-selection tool with multiple backends: Stockfish, the nine Maia human-imitation
  nets (1100–1900), and full Leela.
- **Why:** Human-like move prediction is useful for opponent modelling and coaching, distinct from
  engine-best.
- **Impact:** Per-rating "what would a human play here?" *(Shipped with two latent bugs that made the
  Maia/Leela backends non-functional — see [Bug #5](#bug-5--engine_move-runs-maia-with-a-time-limit)
  and [Bug #7](#bug-7--engine_move-maialeela-never-returned-a-move).)*

### 🆕 `#32` — `batch_review`
- **What:** Analyzes a multi-game export and aggregates win/draw/loss rates, average centipawn loss,
  and recurring blunders, grouped by opening / pawn structure / color.
- **Why:** Surfaces patterns across many games that a single-game review can't.
- **Impact:** "Which openings am I actually losing in, and why?" *(Aggregation math was later
  corrected — see [Bug #2](#bug-2--batch_review-avg_cpl) and
  [Bug #3](#bug-3--batch_review-win-rate-without-a-username).)*

### 📦 `#27` — Installable package (src layout) + distribution scaffolding
- **What:** Repackaged the server as an installable `chess_mcp` package (src layout), added console
  scripts (`chess-mcp`, `chess-files`), a PyPI-publish CI job (OIDC trusted publisher), a
  `smithery.yaml`, and README install instructions.
- **Why:** Make the server `pip install chess-mcp` / `uvx chess-mcp`-able and registry-listable.
- **Impact:** Distribution-ready. **Note:** the *release itself is not yet published* and the
  registries are not yet listed (see [Known remaining work](#known-remaining-work)).

## Phase 2 — Validation & bug fixes (PM)

The morning's code was treated as needing a real correctness pass before being trusted: host tests
are engine-free, several features came from parallel agents, and integration had already caught
shipped bugs. The pass found **seven genuine bugs**.

| # | Bug | Tool | Severity | Commit |
|---|---|---|:---:|---|
| 1 | 5-valued WDL mis-mapped; wins/losses reported as draws | `tablebase_lookup` | 🟧 High | `feb378d` |
| 2 | `avg_cpl` was the mean of the top-3 worst moves, both colors | `batch_review` | 🟥 High | `feb378d` |
| 3 | Win/draw/loss showed misleading zeros without a username | `batch_review` | 🟨 Med | `feb378d` |
| 4 | Username injected into the request URL path (SSRF-ish) | `lichess_games` / `chesscom_games` | 🔒 High | `21e637c` |
| 5 | Maia searched by time → drifted off its target rating | `engine_move` | 🟨 Med | `c8c9aa1` |
| 6 | Drill lists split transposed positions by clock-bearing FEN | `repertoire_vs_history` | 🟨 Med | `a3daacc` |
| 7 | `WeightsFile` passed wrong → Maia/Leela **never ran** | `engine_move` | 🟥 High | `7d5c7cd` |

### Bug 1 — `tablebase_lookup` WDL mapping
- **Root cause:** The 3-valued map was internally inconsistent (`cursed-win → 2` *ignored* the
  50-move rule while `blessed-loss → 0` *respected* it — mirror images, opposite conventions) and
  dumped `maybe-win` / `syzygy-win` / `maybe-loss` / `syzygy-loss` into an `else → 0` branch.
- **Impact:** A genuinely **winning or losing** 7-piece position could be reported as a **draw**,
  defeating the tool's stated purpose ("verify engine conclusions in 7-piece zones").
- **Fix:** A 5-valued Syzygy map (`_WDL_BY_CATEGORY`): `win`/`maybe-win`/`syzygy-win → 2`,
  `cursed-win → 1`, `draw → 0`, `blessed-loss → -1`, `loss`/`maybe-loss`/`syzygy-loss → -2`,
  unknown → `null`. The verbatim `category` is retained for the exact nuance.

### Bug 2 — `batch_review` avg_cpl
- **Root cause:** `avg_cpl` was computed from `get_game_summary`'s `worst_moves`, which is only the
  **top-3** moves — and spans **both colors**.
- **Impact:** The headline "average centipawn loss" was neither an ACPL (grossly inflated) nor
  user-relative (a clean game vs a blundering opponent showed *your* CPL as huge).
- **Fix:** Compute a true ACPL over the **user's own moves** from one `analyze_game(min_cp_loss=0)`
  pass per game; scope `top_blunders` the same way; with a username, restrict to the user's games.

### Bug 3 — `batch_review` win rate without a username
- **Root cause:** Without a username, decisive games resolved to `result=None` and went uncounted,
  so only draws were tallied.
- **Impact:** `win_rate`/`loss_rate` were a misleading `0` that didn't sum to 1.
- **Fix:** `_aggregate_games(records, decided=...)` omits win/draw/loss and worst/best group when
  there is no user POV, leaving `games` + `avg_cpl` + `top_blunders`.

### Bug 4 — Username URL injection 🔒
- **Root cause:** The username was interpolated straight into the URL **path**
  (`...format(user=username)`), and httpx parses `/` and `?` in a string URL as structure.
- **Impact:** A username of `../../../account` retargeted the request to a **different Lichess
  endpoint**, and `evil?max=...` injected query parameters — confirmed against httpx URL parsing.
- **Fix:** `urllib.parse.quote(username, safe="")` on both URL builders; regression test added.
  (FEN-bearing calls already used httpx `params=` and were safe.)

### Bug 5 — `engine_move` runs Maia with a time limit
- **Root cause:** Every backend searched by **wall-clock time**. Maia is human-like only at a single
  node (its raw policy head); a time/depth search runs multi-node PUCT over the Maia net.
- **Impact:** Strength climbed **above** the requested human rating and the move drifted toward
  engine-best — defeating the human-imitation use case.
- **Fix:** `maia-*` now uses `chess.engine.Limit(nodes=1)` (time ignored); Stockfish and full Leela
  still search by time.

### Bug 6 — `repertoire_vs_history` drill-list transposition split
- **Root cause:** `player_deviations` and `uncovered_opponent_moves` were aggregated by **full FEN**,
  whose halfmove/fullmove clocks differ between move orders that transpose to the same position.
- **Impact:** One recurring deviation/gap split into several `count: 1` entries — undercounting and
  fragmenting the exact "drill list" the tool exists to surface, despite its transposition-aware
  contract.
- **Fix:** Aggregate by position (`_position_key_from_fen` — placement + turn + castling + en
  passant), the clock-free identity the in-book walk already uses.

### Bug 7 — `engine_move` Maia/Leela never returned a move
- **Root cause:** The backend opened lc0 with
  `popen_uci(path, options={"WeightsFile": ...})`, but `popen_uci` has **no `options` kwarg** — it
  forwarded `options` to `subprocess.Popen`, raising `TypeError`.
- **Impact:** **Every** `maia-*` and `leela` call returned `backend_error` and produced **no move**.
  The feature had never worked end-to-end. (Stockfish was unaffected — no weight file.) It went
  unnoticed because no test ever ran with real weights, and the mock swallowed the bad kwarg.
- **Fix:** Set the net via `engine.configure({"WeightsFile": ...})` after opening. Verified live
  against a from-source lc0 + maia-1500 build.

## Cross-cutting lesson

Three of the seven bugs shipped **with tests that enshrined the wrong behavior** — the tablebase WDL
tests asserted the inconsistent mapping, and the `engine_move` Maia mock both swallowed and asserted
the broken `options` kwarg. A green suite therefore proved nothing for those paths. The bugs only
surfaced under **real engines, the live API, and actual weights** — which is exactly why the
validation pass ran everything in Docker rather than trusting the host mocks.
